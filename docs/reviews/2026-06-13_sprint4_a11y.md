# A11y pass — Apply chat + Career page (Sprint 4 / D4)

**Date:** 2026-06-13 · **Scope:** `Apply/ApplyPage.jsx`, `Careers/CareerPage.jsx`

> **Method caveat.** axe-core needs a rendered DOM and the dev stack can't boot in this
> sandbox, so this is a **static audit against axe / WCAG 2.1 AA rules** (read the JSX,
> check each rule), not a live `axe-core` run. Re-run axe-core in the browser before sign-off
> (`@axe-core/playwright` in the existing E2E harness is the natural home — see "How to run live").
> Quick, unambiguous fixes were applied now; judgement-call items are left open below.

## Fixed in this pass

| # | Surface | Rule (axe id) | Issue | Fix |
|---|---|---|---|---|
| 1 | Apply | `label` | Chat composer `<textarea>` had only a placeholder, no label | Added `aria-label="Type your reply"` |
| 2 | Apply | `label` | Hidden resume `<input type=file>` had no accessible name | Added `aria-label="Upload resume file"` |
| 3 | Apply | `label` | Hidden video `<input type=file>` had no accessible name | Added `aria-label="Upload video introduction file"` |
| 4 | Apply | `status` / live region | "Typing" animation was silent to screen readers | `role="status"` + `aria-label="Assistant is typing"`; dots `aria-hidden` |
| 5 | Career | `dialog-name` / `aria-dialog` | Apply modal was a plain `<div>` (not a dialog) | `role="dialog"`, `aria-modal="true"`, `aria-labelledby` → titled `<h3 id>` |
| 6 | Career | `button-name` | Modal close button was icon-only (no text) | Added `aria-label="Close"` |
| 7 | Career | `label` | Modal Name/Email labels weren't programmatically associated | Added `htmlFor`/`id` on both fields |

## Open findings (need a decision or a live run)

| # | Severity | Surface | Rule | Finding | Recommendation |
|---|---|---|---|---|---|
| A | **High** | Career | `color-contrast` (1.4.3) | The career page renders **user-chosen** `career_primary_color` as a background under white text (apply CTAs, job-card CTA, match badge) and as text on tinted chips. A light brand color → fails AA contrast. | In `CareerBrandTab`, validate the chosen color's contrast and either warn or auto-pick black/white foreground per luminance. Systemic — affects every branded page. |
| B | Medium | Apply | `focus` / `aria-modal` | The apply chat is a full-page flow (no modal), but the **Career** apply modal doesn't trap focus or restore it on close. | Add focus trap + return focus to trigger; or reuse a shared accessible modal primitive. |
| C | Low | Apply | `image-alt` / decorative | Emoji avatars (🤖, user initial) and emoji inside buttons are announced verbatim by SRs. | Wrap purely-decorative emoji in `aria-hidden`; keep button text labels. |
| D | Low | Apply | `progressbar` | `ApplyProgress` stepper is visual-only (no `role="progressbar"`/`aria-valuenow`). | Add progressbar semantics (the new `OnboardingChecklist` already does this). |
| E | Low | Both | `heading-order` | Verify no skipped heading levels once rendered (static read looks OK; confirm live). | Confirm in the live axe-core run. |

## How to run live (recommended next)

Add an axe assertion to the Playwright E2E suite (already wired via `make e2e`):

```js
// e2e/a11y.spec.ts
import AxeBuilder from '@axe-core/playwright'
test('apply chat has no critical a11y violations', async ({ page }) => {
  await page.goto(applyUrl)                 // seed via /dev/seed-core-loop
  const r = await new AxeBuilder({ page }).withTags(['wcag2a','wcag2aa']).analyze()
  expect(r.violations.filter(v => v.impact === 'critical')).toEqual([])
})
```

Run both surfaces (apply chat + `/careers/:slug`) and triage findings A–E above.
