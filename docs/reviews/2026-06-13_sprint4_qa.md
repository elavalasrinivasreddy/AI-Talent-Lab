## #2 — Career-page branding ✅

**Path:** `Settings/tabs/CareerBrandTab.jsx` → `GET /settings/org` (load) / `PATCH /settings/org` (save) → `SettingsService.update_org` → `OrgRepository.update` (dynamic SET) / `OrgRepository.get_by_id` (`SELECT *`) → `organizations.career_primary_color|career_banner_url|career_tagline` (migration `db/migrations.py:826-829`) → public `GET /careers/{slug}` (selects all three) → `Careers/CareerPage.jsx` applies them.

**Verified:**
- DB columns exist; `OrgProfileUpdate` model carries the three fields (`models/settings.py:26-28`).
- PATCH gated by `require_org_head` (correct — only admins edit branding).
- `CareerPage` consumes `org.career_primary_color` (hero gradient, buttons, chips, job cards, match badges, apply CTA), `career_banner_url` (hero background image), `career_tagline` (hero title). Falls back to brand teal / gradient / `Join {name}` when unset.
- `SettingsLivePreview` wired via `onPreviewUpdate` — live preview reflects edits before save.

**Minor findings (non-blocking):**
1. `CareerBrandTab` sends `null` for cleared fields, but PATCH uses `model_dump(exclude_none=True)` → a field **cannot be reset to empty** once set (the `null` is dropped before the UPDATE).
2. Save failures are swallowed (`catch {}` with a "future iteration" TODO) — no error toast.
3. `career_banner_url` is free-text interpolated into a CSS `url(...)` background; low risk, no validation (by design — external URL, no upload).

## #7 — Audit-log UI ✅

**Path:** `Settings/tabs/AuditTab.jsx` → `GET /settings/audit-logs?limit&offset&search` → `AuditService.get_logs` → returns `{ total, logs: [...] }`.

**Verified:**
- Route gated by `require_org_head`.
- Backend row fields (`id, action, entity_type, entity_id, ip_address, created_at, user_name, user_email`) exactly match what the table renders.
- Search is debounced (300 ms) and backend search spans name/email/action/IP/entity/date.
- Pagination wired (`total > limit`); CSV export is **current page only** and labeled accurately ("Export Page to CSV").

**Minor findings (non-blocking):**
1. UI exposes free-text `search` only; backend also supports `user_id` and `action` filter params that have no UI control yet.
