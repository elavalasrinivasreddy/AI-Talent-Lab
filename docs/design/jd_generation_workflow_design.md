## Problem Statement

The JD generation flow is the heart of AI Talent Lab. It currently works for the
happy path (hire request → HR pickup → JD → team lead approval → position open) but
breaks or leaves gaps in at least 12 documented edge cases: no-department orgs, direct
hire (Flow 2) reviewer assignment, feedback injection on rejection, chat deletion policy,
save-draft gating, same-title collisions, admin modify-on-approve, cancel-with-notes,
and what "read-only" means at each stage.

This document maps both position creation flows end-to-end, resolves all edge cases,
and provides a clean implementation blueprint.

---

## The Two Flows

### FLOW 1 — Hire Request Path (Team Lead initiated)

```
Team Lead → Hire Request
     ↓
Dept Admin → Approve / Reject / Approve-with-changes+notes
     ↓ (if approved)
HR Pickup (dept-scoped or global)
     ↓
JD Chat: intake → internal_check → market_research → jd_variants → final_jd → bias_check
     ↓ (bias check done)
HR: Finalize JD → ATS config popup → Submit for Team Lead Approval
     ↓
Chat + position: READ-ONLY from this point
     ↓
Team Lead → Review JD → Approve / Reject-with-notes
     ↓ (if approved)
Position: OPEN → candidates can apply → chat permanently locked
     ↓ (if rejected)
HR gets notification + notes visible in JD tab + notes auto-injected in chat
HR: resumes chat → revision agent updates JD sections (no full regen)
Bias check resets → HR re-checks → Finalize → Submit again → same approval loop
```

### FLOW 2 — Direct New Hire (HR/Admin/Head initiated)

```
User clicks "New Hire" tab
     ↓
JD Chat: same stages as Flow 1 (intake asks role, skills, details)
     ↓ (bias check done)
User: Finalize JD → ATS config popup → Submit for Review
     ↓
Reviewer resolved by rules (see Department Matrix below)
     ↓
If creator = org_head → position goes OPEN directly, no approval step, chat permanently locked
If reviewer exists → same approval loop as Flow 1
```

## Status Nomenclature (IMPORTANT — prevents label collision)

Two entities have statuses. They must be named distinctly.

### hire_request.status

```
draft → submitted → admin_reviewing → approved | approved_modified | rejected | cancelled
```

`hr_pickup` is NOT a hire request status. Once HR picks up, the hire request stays
`approved` — the pickup is tracked on the position.

### position.status

```
draft → jd_in_progress → pending_jd_approval → draft_needs_revision → approved → open
```

`hr_pickup` is retired. The new value is `jd_in_progress` for clarity.

## Hire Request State Machine — Complete

### States
```
draft → submitted → admin_reviewing → approved | approved_modified | rejected | cancelled
```

`admin_reviewing` is a stable intermediate state set when the admin opens the review
modal. This prevents two admins from simultaneously reviewing the same request.

### Transitions

| From | Action | By | To | Notes required |
|---|---|---|---|---|
| draft | Submit | team_lead | submitted | — |
| submitted | Cancel | team_lead | cancelled | YES — shown to admin if admin_reviewing |
| submitted | Begin review | dept_admin | admin_reviewing | — |
| admin_reviewing | Approve | dept_admin | approved | — |
| admin_reviewing | Approve + modify | dept_admin | approved_modified | YES — diff shown to team_lead |
| admin_reviewing | Reject | dept_admin | rejected | YES — shown to team_lead |
| admin_reviewing | Release lock | dept_admin | submitted | — (admin closed without action) |
| approved | Cancel | team_lead | cancelled | YES — all HRs notified |
| approved_modified | Cancel | team_lead | cancelled | YES |
| approved / approved_modified | HR picks up | hr | position.status = jd_in_progress | — |

**Team lead cannot cancel once an HR has started jd_in_progress.**
At that stage, they must reject at the JD review step.

**Notes visibility:**
- Admin rejection notes → team lead hire request view
- Admin modification notes → team lead hire request view (with field diff)
- Team lead cancellation notes → admin notification + hire request view

## Position State Machine — Complete

### States
```
draft → jd_in_progress → pending_jd_approval → draft_needs_revision → approved → open
```

Flow 2 also starts at `draft`, skips hire request linkage.

### Transitions

| From | Action | By | To | Notes required |
|---|---|---|---|---|
| draft | Hire request approved + HR picks up | hr | jd_in_progress | — |
| draft | HR/admin creates direct (Flow 2) | creator | jd_in_progress | — |
| jd_in_progress | Submit for approval | hr/admin | pending_jd_approval | — |
| jd_in_progress | org_head submits (Flow 2) | org_head | open | — (bypass) |
| pending_jd_approval | Reviewer approves | reviewer | approved | — |
| approved | Auto-publish | system | open | — |
| pending_jd_approval | Reviewer rejects | reviewer | draft_needs_revision | YES |
| draft_needs_revision | HR re-submits | hr | pending_jd_approval | — |
| open | — | — | open | Immutable |

**approved → open is immediate** (no separate publish step in Phase 1).

## Department Matrix — Who Picks Up / Who Reviews

### HR Pickup Visibility (Flow 1)

| Position's department | Who sees it in pickup queue |
|---|---|
| Has a department | HRs in that department + HRs with no department (global HRs) |
| No department set | All HRs in the org |

Global HRs (no department assigned) see ALL requests across all departments.
This is intentional for small orgs where HR is org-wide. Configurable to restrict
in Approval Rules (Phase 2).

### Flow 2 Reviewer Resolution

| Who creates | Department situation | Reviewer | Notes |
|---|---|---|---|
| `org_head` | any | **No review** — position goes live immediately | Org head has full authority |
| `dept_admin` | has department | `org_head` | Configurable OFF in Approval Rules |
| `dept_admin` | no department (global) | `org_head` | Same rule |
| `hr` | has department + dept has admin | `dept_admin` of that department | Standard path |
| `hr` | has department + dept has NO admin | `org_head` (escalated) | Warn in FinalJDCard |
| `hr` | no department (global HR) | `org_head` | Warn in FinalJDCard |

**Fallback chain:** If the resolved reviewer doesn't exist or is the same person as
the creator → escalate one level → if still unresolvable → org_head with a flag.

**Configurable via Approval Rules settings (Phase 1):**
- "Direct hire positions created by HR require review" → toggle (default: ON)
- "Direct hire positions created by Dept Admin require org_head review" → toggle (default: ON)

### For Orgs With No Departments

| Creator | Reviewer |
|---|---|
| `org_head` | No review |
| `dept_admin` | `org_head` |
| `hr` | `org_head` (since no dept_admin exists) |

## JD Chat Lifecycle — Status + Permissions at Each Stage

| Chat stage | position.status | Chat R/W | Finalize enabled | Save Draft | Delete session |
|---|---|---|---|---|---|
| intake → bias_check in progress | jd_in_progress | Read-Write | NO | YES | YES |
| bias_check DONE (not reset) | jd_in_progress | Read-Write | YES | YES | YES |
| Submitted for approval | pending_jd_approval | READ-ONLY | — | — | NO |
| Approved | approved / open | READ-ONLY | — | — | NO |
| Rejected — needs revision | draft_needs_revision | Read-Write | NO (bias reset) | YES | YES |
| Cancelled position | cancelled | — | — | — | YES |

**Save Draft:** Always available. Not gated. It's a draft — let people save.

**Finalize JD:** Enabled only after bias check run in current working session.
"Current working session" = since last JD revision OR since chat opened on a fresh position.

**Bias check reset triggers:**
- AI performs any `rewrite_section` action
- HR manually edits JD text in WYSIWYG mode
- Any revision agent action fires

## Rejection → Revision Flow

### Problem
Notes visible in JD tab as an amber banner. Notes NOT in the chat window.
HR must manually read notes and type a prompt.

### Solution

When HR opens "Resume AI Chat" after `draft_needs_revision`:

**Step 1 — Backend injects a feedback message into the session.**

On session load (`GET /chat/sessions/{session_id}`), if `position.approval_status = changes_requested`:
- Check if a `feedback_injection` message already exists for this revision cycle
- If not: `INSERT INTO chat_messages (session_id, role='system', message_type='feedback_injection', content=review_notes, created_at=now())`
- Return this message as part of the messages array

Storage schema:
```sql
-- Existing chat_messages table, new message_type value
INSERT INTO chat_messages (
  session_id, role, content, message_type, created_at
) VALUES (
  $1, 'system', $review_notes, 'feedback_injection', NOW()
) ON CONFLICT (session_id, message_type, revision_cycle) DO NOTHING;
```

The `ON CONFLICT` guard prevents double-injection if HR opens the chat multiple times.
A `revision_cycle` counter on the position tracks which rejection cycle we're on (1, 2, 3...).

**Step 2 — Frontend renders it as a distinct UI card in the message list.**

```
┌─ Team Lead Feedback ─────────────────────────────────┐
│ Arun (Team Lead) · 2 hours ago                        │
│                                                        │
│ "The JD mentions Python as optional but we need it   │
│  as mandatory. Also, salary band is too wide."        │
│                                                        │
│ Respond below or ask AI to apply these changes.       │
└────────────────────────────────────────────────────────┘
```

**Step 3 — Revision agent behavior when HR replies.**

- If feedback is about skills → extract → `UPDATE positions SET required_skills = ...`
- If feedback is about JD sections → `rewrite_section` for targeted paragraphs
- Never full regen unless HR explicitly says "regenerate from scratch"
- After revision: bias check banner reappears

### Deleted Session Edge Case

If HR deleted the session AND `draft_needs_revision`:
- New session created on first message
- Backend injects feedback message into the NEW session at creation time (same logic)
- JD content is on `positions.final_jd` — not on the session — so revision agent can still work

## Dynamic Chat Greeting

Three contexts, three greetings. All composed on the frontend from `AuthContext` + `location.state`.
No backend change needed.

```js
function buildGreeting(user, locationState, sessionData) {
  const firstName = user?.name?.split(' ')[0] || 'there'

  // Hire request pickup
  const req = locationState?.hireRequest
  if (req?.role_name) {
    const from = req.requested_by_name ? ` from ${req.requested_by_name}` : ''
    return `Hi ${firstName}! I see you're picking up the ${req.role_name} request${from}. Let's build a great JD.`
  }

  // Resuming after rejection
  if (sessionData?.position_approval_status === 'changes_requested') {
    const reviewer = sessionData.reviewer_name || 'the reviewer'
    return `Welcome back, ${firstName}. ${reviewer} has reviewed the JD and left feedback. Let's work through the changes.`
  }

  // Resuming an existing session
  if (sessionData?.id) {
    const title = sessionData.title || 'your JD'
    return `Welcome back, ${firstName}! Let's continue working on ${title}.`
  }

  // Fresh new hire
  return `Hi ${firstName}! What role are you looking to fill today?`
}
```

`ChatContext.jsx`: Replace the `GREETING` constant with `buildGreeting(user, locationState, null)`.
Pass `sessionData` once `loadSession` completes.

## Post-Approval: What Happens (Phase 1)

When reviewer approves a position:
1. `position.status → open`
2. `position.approval_status → approved`
3. Career page: auto-published if org has CareerBrand configured (existing logic)
4. ATS sourcing: starts at configured frequency (existing logic)
5. Chat session: permanently locked — delete disabled, read-only enforced
6. Hire request (if Flow 1): `hire_request.status → fulfilled`
7. Notifications: HR + team lead notified "Position is now open"

**After open, nothing is editable except:**
- ATS threshold (operational tuning, not a JD change)
- Sourcing frequency (operational)

### CRITICAL BLOCKERS (must fix before any code)

---

#### C1 — HR Pickup Has No Atomic Lock (Race Condition)

**Failure:** Two HRs in the same department both see the request, both click "Pick Up"
simultaneously. Both pass the status check, both create sessions → duplicate `jd_in_progress`
positions, double work, or a DB constraint crash.

**Fix:** Atomic compare-and-swap at the DB level:
```sql
UPDATE positions
SET status = 'jd_in_progress', picked_up_by = $hr_id, picked_up_at = NOW()
WHERE id = $position_id
  AND status = 'draft'
  AND picked_up_by IS NULL
RETURNING id;
```
Zero rows returned → "Already picked up by [name]. View position →". Never show an error — show who has it.

Backend: `position_service.pickup()` must use this pattern. Never a SELECT then UPDATE.

---

#### C2 — `admin_reviewing` Lock Has No TTL or Takeover

**Failure:** Admin opens review modal → tab crashes / loses network / leaves for lunch.
Request is locked in `admin_reviewing` forever. No other admin can act. Team lead sees
"under review" with no path forward and no visible owner.

**Fix:**
- Add `reviewing_locked_at TIMESTAMPTZ` and `reviewing_locked_by INT` columns to `hire_requests`
- Lock TTL: 30 minutes. After TTL, any admin can take over
- Cron / background task: `UPDATE hire_requests SET status='submitted', reviewing_locked_by=NULL WHERE status='admin_reviewing' AND reviewing_locked_at < NOW() - INTERVAL '30 minutes'`
- UI: Show "Being reviewed by Priya since 14 min ago. [Take over]" to other admins when TTL > 10 min elapsed
- Takeover: atomic `UPDATE ... WHERE reviewing_locked_by = $old_admin_id`

---

#### C3 — Reviewer Deletion Causes Permanent Dead-End

**Failure:** Reviewer is resolved and stored at submit time. If that user is deleted
while position is `pending_jd_approval`, no one has review authority. HR sees READ-ONLY
chat with no path forward. Position stuck permanently.

**Fix:**
- On `users DELETE` (or deactivate): re-run reviewer resolution for all positions where
  `reviewer_id = deleted_user_id AND status = 'pending_jd_approval'`
- Assign the new resolved reviewer; notify them
- If no reviewer can be resolved → assign to org_head + flag for admin attention
- HR sees: "Your reviewer was removed. The position has been reassigned to [new reviewer]."
- Same trigger for `departments DELETE`: re-resolve all positions in that dept

---

#### C4 — ATS Submit and Post-Approval Side Effects Are Not Transactional

**Failure (ATS popup):** HR fills ATS config, submits. Call fails after ATS config is
saved but before `position.status` updates → config saved, status stays `jd_in_progress`.
HR re-opens popup: is config pre-filled or blank? Race condition.

**Failure (post-approval):** 6 sequential operations on approval (status, approval_status,
career publish, ATS start, chat lock, hire_request fulfilled). If step 3 fails,
steps 1–2 already committed → position is `open` but request not `fulfilled`.

**Fix (ATS submit):**
- Save ATS config + status transition in one transaction:
  ```sql
  BEGIN;
    UPDATE positions SET ats_threshold=$t, sourcing_freq=$f WHERE id=$id;
    UPDATE positions SET status='pending_jd_approval' WHERE id=$id;
  COMMIT;
  ```
- Frontend: preserve popup form state in `sessionStorage` on failure; auto-restore on reopen

**Fix (post-approval):**
- Transaction covers: `position.status = open`, `position.approval_status = approved`,
  `hire_request.status = fulfilled` (if Flow 1)
- External side effects (career page publish, ATS sourcing start) run as **async jobs after commit**
- If external jobs fail: retry queue; notify org_head if still failing after 3 retries
- Never let external failure roll back a legitimate approval

---

#### C5 — Self-Approval and Cross-Dept Authority Not Enforced Server-Side

**Failure (self-approval):** The `/resolve-reviewer` preview shows `is_self_approve: true`
as a warning, but nothing in the spec says the *submit endpoint* enforces it. A crafted
API call could set `reviewer_id = creator_id` and self-approve.

**Failure (cross-dept):** The approve endpoint likely only checks role (`is team_lead`),
not scope. A team_lead from Dept A could approve a Dept B position.

**Fix:**
- `POST /positions/{id}/submit-for-approval`:
  - Re-run reviewer resolution server-side (do NOT trust client-supplied reviewer_id)
  - Assert `resolved_reviewer_id != current_user.id` — if equal, escalate or 403
- `POST /positions/{id}/approve`:
  - Assert `current_user.id == position.reviewer_id`
  - Assert reviewer's current role and dept_scope still grant authority over this position's dept
  - Snapshot the role at approval time: store `approved_by_role` on the position
- Org_head bypass: verify `current_user.role == 'org_head'` server-side at submit, not via flag

### STATE MACHINE FIXES

#### S1 — `approved` is a Phantom State

`pending_jd_approval → approved → open` with "auto-publish" creates a state that
appears and disappears in the same transaction. Nobody can observe it; it adds no value.

**Fix:** Remove `approved` from Phase 1. Transition directly:
```
pending_jd_approval → open  (on reviewer approval)
```
Update the position machine table accordingly. `position.approval_status` stays as
`approved` (separate field) so the approval fact is recorded without polluting the
lifecycle status.

#### S2 — `approved_modified` Hire Request Cannot Be Picked Up

Pickup guard was `status = 'approved'` — excludes `approved_modified`.

**Fix:** Guard: `status IN ('approved', 'approved_modified')`. Apply in all pickup
queries and the `picked_up_by` CAS update above.

#### S3 — `cancelled` Position State Is Unreachable

No transition leads to `position.status = 'cancelled'`. Hire request cancellation
cancels the *request*, not the position. A `jd_in_progress` position whose request
is cancelled is an orphan.

**Fix:** Add transition:
```
jd_in_progress → cancelled
```
Triggered by: hire request cancelled by team lead after pickup (currently blocked).
Decision: allow team lead to cancel after pickup IF the JD has not been submitted yet.
Once `pending_jd_approval`, team lead cannot cancel — they must reject.

Updated cancel authority:

| Position status | Team lead can cancel? |
|---|---|
| draft (request approved, not picked up) | YES |
| jd_in_progress | YES, with mandatory notes |
| pending_jd_approval | NO — must reject |
| open | NO |

#### S4 — Infinite Rejection Loop Has No Bound

A team lead can reject → HR revises → resubmits → reject → repeat indefinitely.
No escalation, no cap.

**Fix:** Track `revision_cycle INT` on position. After 3 rejections, org_head is
notified and added as a co-reviewer. This is a Phase 2 feature; Phase 1 must at
minimum persist `revision_cycle` so Phase 2 can use it.

---

### CONCURRENCY FIXES

#### CC1 — Same-Title Check Is TOCTOU

Two concurrent intakes for "Senior Engineer" both pass the SELECT check and both create.

**Fix:** The suffix "(2)" must be computed inside a transaction using `SELECT ... FOR UPDATE`
or a unique partial index with a sequence. Alternatively: enforce uniqueness at a higher
level (position title + org + active status) and handle the collision gracefully on insert.

#### CC2 — Feedback Injection ON CONFLICT Requires Non-Existent Constraint

`ON CONFLICT (session_id, message_type, revision_cycle)` will fail at runtime because:
- No unique constraint exists on `chat_messages` for this tuple
- `revision_cycle` is not a column on `chat_messages`

**Fix:**
- Add `revision_cycle INT` to `chat_messages` (or use `message_metadata JSONB`)
- Add migration: `CREATE UNIQUE INDEX uq_chat_feedback_injection ON chat_messages (session_id, message_type, revision_cycle) WHERE message_type = 'feedback_injection';`
- Alternatively: check existence before insert (`SELECT 1 FROM chat_messages WHERE session_id=$1 AND message_type='feedback_injection' AND revision_cycle=$2`) inside a transaction

```
draft
  ↓ HR picks up (CAS, atomic)

  ↓ HR submits (transactional)      → pending_jd_approval

pending_jd_approval
  ↓ Reviewer approves               → open
  ↓ Reviewer rejects                → draft_needs_revision
draft_needs_revision
  ↓ HR re-submits                   → pending_jd_approval
open
  (immutable — no transitions out in Phase 1)
cancelled
  (terminal)
```
