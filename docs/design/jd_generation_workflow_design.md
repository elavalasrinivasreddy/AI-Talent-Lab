## Problem Statement

The JD generation flow is the heart of AI Talent Lab. Two position creation paths exist.
Both share the same JD generation pipeline but diverge in reviewer assignment. The current
implementation covers the happy path; this document specifies every state, transition,
edge case, failure mode, and security rule needed to ship it correctly.

## Flows at a Glance

### FLOW 1 — Hire Request Path (Team Lead initiated)

```
Team Lead → Hire Request (submitted)
     ↓
Dept Admin → admin_reviewing → approved | approved_modified | rejected
     ↓ (if approved or approved_modified)
HR Pickup — atomic CAS, dept-scoped visibility
     ↓
position: jd_in_progress
JD Chat: intake → internal_check → market_research → jd_variants → final_jd → bias_check
     ↓ (bias done, Finalize enabled)
HR: Finalize JD → ATS config popup → Submit for Team Lead Approval
     ↓
position: pending_jd_approval — chat READ-ONLY
     ↓
Team Lead → Approve | Reject-with-notes
     ↓ (approved)
position: open — chat permanently locked — career page published — ATS starts
     ↓ (rejected)
position: draft_needs_revision
HR: notification + notes injected in chat → revision → bias resets → re-submit → same loop
```

### FLOW 2 — Direct New Hire (HR / Dept Admin / Org Head)

```
User clicks "New Hire"
     ↓
position: jd_in_progress (created immediately — no draft step)
JD Chat: same pipeline as Flow 1
     ↓ (bias done)
User: Finalize JD → ATS config popup → Submit for Review
     ↓
Reviewer resolved server-side (see Department Matrix)
     ↓
If creator = org_head → position: open immediately (bypass, no approval)
If reviewer exists → pending_jd_approval → same approval loop as Flow 1
```

## Status Nomenclature — Two Entities, No Label Collision

### hire_request.status

```
draft → submitted → admin_reviewing → approved | approved_modified | rejected | cancelled
```

`hr_pickup` is NOT a hire request status. Once HR picks up, the hire request stays
`approved` or `approved_modified`. Pickup is tracked on the position via `picked_up_by`.

### position.status

```
jd_in_progress → pending_jd_approval ⇄ draft_needs_revision → open
jd_in_progress → cancelled
```

`hr_pickup` retired. `approved` removed — approval goes directly to `open`
(the fact is recorded in `position.approval_status`, not duplicated in `position.status`).
Flow 2 positions are created directly in `jd_in_progress` — there is no `draft` step.

## Hire Request State Machine

### States

```
draft → submitted → admin_reviewing → approved | approved_modified | rejected | cancelled
```

`admin_reviewing` is a stable state set atomically when an admin opens the review modal.
It prevents two admins from colliding. It carries a 30-minute TTL.

### Transitions

| From | Action | By | To | Notes required |
|---|---|---|---|---|
| draft | Submit | team_lead | submitted | — |
| submitted | Cancel | team_lead | cancelled | YES — shown to admin if lock active |
| submitted | Begin review (atomic) | dept_admin | admin_reviewing | — |
| admin_reviewing | Approve | dept_admin | approved | — |
| admin_reviewing | Approve + modify fields | dept_admin | approved_modified | YES — diff stored + shown to TL |
| admin_reviewing | Reject | dept_admin | rejected | YES — shown to team_lead |
| admin_reviewing | Close without action | dept_admin | submitted (lock released) | — |
| admin_reviewing | TTL expired (30 min) | system | submitted (lock auto-released) | — |
| approved | Cancel (before HR pickup) | team_lead | cancelled | YES — dept HRs notified |
| approved_modified | Cancel (before HR pickup) | team_lead | cancelled | YES |
| approved / approved_modified | HR picks up (atomic CAS) | hr | hire_request stays; position → jd_in_progress | — |

### admin_reviewing Lock Rules

Columns: `reviewing_locked_by INT FK users(id) ON DELETE SET NULL`, `reviewing_locked_at TIMESTAMPTZ`

Lock set (atomic):
```sql
UPDATE hire_requests
SET status = 'admin_reviewing',
    reviewing_locked_by = $admin_id,
    reviewing_locked_at = NOW()
WHERE id = $id AND status = 'submitted'
RETURNING id;
-- Zero rows → another admin beat you. Show who has it.
```

TTL release (background job or trigger):
```sql
UPDATE hire_requests
SET status = 'submitted', reviewing_locked_by = NULL, reviewing_locked_at = NULL
WHERE status = 'admin_reviewing'
  AND reviewing_locked_at < NOW() - INTERVAL '30 minutes';
```

Takeover UI: after 10 min elapsed, other admins see "Being reviewed by [name] (14 min ago) — Take over".
Takeover uses the same atomic UPDATE (zero rows = someone else already took over).

### Notes Visibility

- Admin rejection notes → team lead hire request view
- Admin modification notes → team lead hire request view with field diff
- Team lead cancellation notes → admin notification + hire request view

## Position State Machine

### States

```
jd_in_progress
  → pending_jd_approval
      → draft_needs_revision
          → pending_jd_approval  (re-submit loop)
      → open  (terminal — approved)
  → cancelled  (terminal)
```

`approved` removed. `draft` removed for Flow 2 (positions created directly in `jd_in_progress`).

### Transitions

| From | Action | By | To | Notes required |
|---|---|---|---|---|
| — | HR picks up hire request (atomic CAS) | hr | jd_in_progress | — |
| — | HR/admin creates direct (Flow 2) | creator | jd_in_progress | — |
| jd_in_progress | TL cancels hire request (before submit only) | team_lead | cancelled | YES |
| jd_in_progress | HR submits (transactional) | hr / admin | pending_jd_approval | — |
| jd_in_progress | org_head submits (Flow 2 bypass) | org_head | open | — |
| pending_jd_approval | HR withdraws submission | hr | jd_in_progress | — |
| pending_jd_approval | Reviewer approves | reviewer | open | — |
| pending_jd_approval | Reviewer rejects | reviewer | draft_needs_revision | YES |
| draft_needs_revision | HR re-submits | hr | pending_jd_approval | — |
| open | — | — | open | Immutable in Phase 1 |
| cancelled | — | — | cancelled | Terminal |

### Atomic HR Pickup (compare-and-swap)

```sql
UPDATE positions
SET status = 'jd_in_progress',
    picked_up_by = $hr_id,
    picked_up_at = NOW()
WHERE id = $position_id
  AND picked_up_by IS NULL
RETURNING id;
```

Zero rows → already picked up. Show: "Picked up by [name]. View position →". Never a generic error.
Hire request pickup guard: `hire_request.status IN ('approved', 'approved_modified')`.

### Team Lead Cancel-After-Pickup Rules

| position.status | TL can cancel? |
|---|---|
| jd_in_progress | YES — mandatory notes; HR notified |
| pending_jd_approval | NO — must reject at review step |
| open | NO |

### HR Withdraw Submission

`pending_jd_approval → jd_in_progress`. Reviewer notified "submission withdrawn."
Bias resets on withdraw. `revision_cycle` does NOT increment (only increments on reviewer rejection).

### Revision Cycle Cap

`revision_cycle INT DEFAULT 0` on positions. Increments on reviewer rejection only.
Phase 1: persist column. Phase 2: after 3 rejections, org_head added as co-reviewer.

### open State — Post-Approval Sequence

When reviewer approves — **single transaction**:
```sql
BEGIN;
  UPDATE positions SET status = 'open', approval_status = 'approved' WHERE id = $id;
  UPDATE hire_requests SET status = 'fulfilled' WHERE id = $hire_request_id; -- Flow 1 only
COMMIT;
```

After commit — **async retryable jobs**:
1. Career page publish (if CareerBrand configured)
2. ATS sourcing start at configured frequency
3. Notifications to HR + team_lead

If async job fails: retry queue. Notify org_head after 3 failed retries.

Post-open editable fields (role-gated: `org_head` and `dept_admin` only):
- `ats_threshold`
- `sourcing_freq`

Everything else is immutable in Phase 1.

## Department Matrix — Pickup & Reviewer Resolution

### HR Pickup Visibility (Flow 1)

| Position's department | Who sees it in pickup queue |
|---|---|
| Has a department | HRs in that department + global HRs (no dept assigned) |
| No department set | All HRs in the org |

Global HRs (no department) see ALL requests. Configurable restriction: Phase 2.

### Flow 2 Reviewer Resolution

Resolution runs **server-side at submit time**. Client-supplied reviewer_id is ignored.
Result stored as `position.reviewer_id`. Self-approve blocked server-side.

| Creator role | Department situation | Resolved reviewer |
|---|---|---|
| `org_head` | any | No reviewer — position goes to `open` directly |
| `dept_admin` | has dept or global | `org_head` (configurable OFF in Approval Rules) |
| `hr` | has dept + dept_admin exists | `dept_admin` of that department |
| `hr` | has dept + no dept_admin | Escalates to `org_head` |
| `hr` | global (no dept) | `org_head` |

**Fallback chain:** If resolved reviewer = creator → escalate one level.
If no valid reviewer exists anywhere → **block submit**:
"No reviewer available. Ask your workspace admin to assign an org head before submitting."

**Configurable via Approval Rules settings:**
- "Direct hires by HR require review" → toggle (default: ON)
- "Direct hires by Dept Admin require org_head review" → toggle (default: ON)

### No-Department Orgs

| Creator | Reviewer |
|---|---|
| `org_head` | No review |
| `dept_admin` | `org_head` |
| `hr` | `org_head` |

### Authority Snapshotting

`submitted_by_role` and `reviewer_role_at_submit` stored on position at submit time.
Role changes mid-flow do not retroactively alter routing or authority.

The approve endpoint verifies:
1. `current_user.id == position.reviewer_id`
2. User still has sufficient role (not demoted below reviewer threshold)
3. User still has dept_scope over this position's department

## JD Chat Lifecycle — Permissions at Each Stage

| position.status | Chat R/W | Finalize enabled | Save Draft | Delete session |
|---|---|---|---|---|
| jd_in_progress (bias not run / reset) | Read-Write | NO | YES | YES |
| jd_in_progress (bias done, not reset) | Read-Write | YES | YES | YES |
| pending_jd_approval | READ-ONLY | — | — | NO (disabled + tooltip) |
| draft_needs_revision | Read-Write | NO (bias reset) | YES | YES |
| open | READ-ONLY | — | — | NO (hidden entirely) |
| cancelled | — | — | — | YES |

**Save Draft:** Always available. Never gated.

**Finalize JD:** Enabled only after bias check runs. Resets when:
- AI executes any `rewrite_section` action
- HR edits JD text in WYSIWYG
- Any revision agent action fires

**Bias reset is pessimistic:** resets *before* the rewrite begins. If the rewrite crashes,
Finalize is already disabled — biased JD cannot ship due to a failed post-rewrite reset.

## ATS Submit — Transactional Contract

```sql
BEGIN;
  UPDATE positions
  SET ats_threshold = $t,
      sourcing_freq = $f,
      status = 'pending_jd_approval',
      reviewer_id = $resolved_reviewer_id,
      submitted_by_role = $current_user_role,
      submitted_at = NOW()
  WHERE id = $position_id;
COMMIT;
```

Single transaction. On failure: keep `jd_in_progress`, preserve ATS config in
`sessionStorage` client-side (auto-restore when HR reopens popup).

API: `GET /api/v1/positions/resolve-reviewer?position_id=X`
Returns: `{ reviewer_id, reviewer_name, reviewer_role, department, is_bypass, warning }`
Called when chat reaches `final_jd` stage (give HR time to read before submit).

## Dynamic Greeting

All composed on frontend from `AuthContext` + `location.state`. No backend change.

```js
function buildGreeting(user, locationState, sessionData) {
  const firstName = user?.name?.split(' ')[0] || 'there'
  const req = locationState?.hireRequest
  if (req?.role_name) {
    const from = req.requested_by_name ? ` from ${req.requested_by_name}` : ''
    return `Hi ${firstName}! I see you're picking up the ${req.role_name} request${from}. Let's build a great JD.`
  }
  if (sessionData?.position_approval_status === 'changes_requested') {
    const reviewer = sessionData.reviewer_name || 'the reviewer'
    return `Welcome back, ${firstName}. ${reviewer} sent the JD back with feedback. Let's work through the changes.`
  }
  if (sessionData?.id) {
    return `Welcome back, ${firstName}! Let's continue working on ${sessionData.title || 'your JD'}.`
  }
  return `Hi ${firstName}! What role are you looking to fill today?`
}
```

Replace `GREETING` constant in `ChatContext.jsx` with `buildGreeting(user, locationState, null)`.
Pass `sessionData` once `loadSession` completes.

## Rejection → Revision Flow

### DB Migration (required before feedback injection)

```sql
ALTER TABLE chat_messages ADD COLUMN IF NOT EXISTS message_type TEXT;
ALTER TABLE chat_messages ADD COLUMN IF NOT EXISTS revision_cycle INT DEFAULT 0;
CREATE UNIQUE INDEX IF NOT EXISTS uq_chat_feedback_injection
  ON chat_messages (session_id, message_type, revision_cycle)
  WHERE message_type = 'feedback_injection';
ALTER TABLE positions ADD COLUMN IF NOT EXISTS revision_cycle INT DEFAULT 0;
```

### Feedback Injection Logic

On `GET /chat/sessions/{session_id}`, if `position.approval_status = 'changes_requested'`:

```sql
INSERT INTO chat_messages
  (session_id, role, content, message_type, revision_cycle, created_at)
VALUES
  ($session_id, 'system', $review_notes, 'feedback_injection', $revision_cycle, NOW())
ON CONFLICT (session_id, message_type, revision_cycle) DO NOTHING;
```

Idempotent — safe to call on every session load. No duplicate banners.

### Feedback Card UI

```
┌─ Team Lead Feedback ─────────────────────────────────┐
│ Arun Mehta (Team Lead) · 2 hours ago                  │
│                                                        │
│ "Python must be mandatory, not optional. Salary      │
│  band is too wide — narrow to 12-15L."                │
│                                                        │
│ Reply below or ask AI to apply these changes.         │
└────────────────────────────────────────────────────────┘
```

### Revision Agent Behavior

- Skills feedback → `UPDATE positions SET required_skills = ...`
- JD text feedback → `rewrite_section` for targeted paragraphs (no full regen)
- Full regen only if HR explicitly says "regenerate from scratch"
- Bias flag resets pessimistically before rewrite begins

### Deleted Session Edge Case

If session deleted AND `draft_needs_revision`: fresh session is created on first message.
Feedback injection fires into the new session using the same idempotent insert.
JD content lives on `positions.final_jd` — revision agent can still work.

## Same Title Handling

After role name captured in intake:

```sql
SELECT id, status, role_name
FROM positions
WHERE org_id = $1
  AND LOWER(role_name) = LOWER($2)
  AND status != 'cancelled';
```

If match found, chat injects a card (not a blocking modal):

```
┌─ Heads up ────────────────────────────────────────────┐
│ "Senior Engineer" already exists (Open).               │
│ [View existing position]  [Continue with new one]      │
└────────────────────────────────────────────────────────┘
```

Suffix computed transactionally to prevent TOCTOU:
```sql
SELECT COUNT(*) + 1 FROM positions
WHERE org_id=$1 AND LOWER(role_name)=LOWER($2) FOR UPDATE;
```

"View existing" → navigate away. Auto-cancel the `jd_in_progress` position after 15 min
if no messages were sent (cleanup job).

## Data Integrity Rules

| Scenario | Rule |
|---|---|
| User deleted while `reviewer_id` on pending position | `ON DELETE SET NULL` + trigger: re-resolve reviewer for all affected positions |
| User deleted while `requested_by` on hire request | `ON DELETE SET NULL`, display as "Former employee" |
| User deleted while `picked_up_by` on position | `ON DELETE SET NULL` — position stays `jd_in_progress`, unassigned |
| User deleted while `reviewing_locked_by` on hire_request | `ON DELETE SET NULL` — auto-releases the review lock |
| Department deleted | Block if active positions exist. Show: "Reassign or close N positions first." |
| `reference_position_id` (same-title link) | `ON DELETE SET NULL` — losing the reference is acceptable |
| Diff JSON for admin modify | Store as `JSONB` with `schema_version: 1` field |

## Security Rules

All enforced server-side. Preview UI is informational only.

| Rule | Enforcement point |
|---|---|
| Self-approval blocked | `submit_for_approval()`: resolve reviewer server-side; assert `resolved != current_user.id`; if equal escalate or 403 |
| Cross-dept review blocked | `approve()`: assert `current_user.id == position.reviewer_id` AND dept_scope matches position's dept |
| Org_head bypass verified | `submit_for_approval()`: re-check `current_user.role == 'org_head'` server-side |
| ATS config editable post-open | Gate on `role IN ('org_head', 'dept_admin')` in the update endpoint |
| Role change mid-flow | Authority snapshotted at submit (`submitted_by_role`, `reviewer_role_at_submit`) — role changes don't retroactively alter routing |
| Global HR data visibility | Known gap — Phase 2 adds visibility restriction toggle in Approval Rules |
